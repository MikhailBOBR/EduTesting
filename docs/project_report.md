# Пояснительная записка к проекту EduTesting

## 1. Анализ предметной области

В образовательном процессе преподавателю необходимо быстро проводить проверку знаний, а студенту получать прозрачную обратную связь. Бумажные формы контроля неудобны, требуют много ручной проверки и не дают гибкой аналитики. Веб-сервис онлайн тестирования решает эту проблему за счёт централизованного доступа к тестам, автоматического оценивания и хранения результатов.

Ключевые сущности предметной области:

- пользователь системы
- преподаватель
- студент
- учебный курс
- тест
- вопрос
- вариант ответа
- попытка прохождения
- ответ на вопрос

Основные бизнес-сценарии:

- преподаватель регистрируется, создаёт курс и публикует тест
- преподаватель публикует объявления и организационные сообщения по курсу
- преподаватель добавляет вопросы и правильные варианты ответов
- студент регистрируется, записывается на курс по публичной ссылке или коду доступа и запускает тест
- преподаватель может выдать студенту индивидуальные условия по конкретному тесту
- система автоматически сохраняет черновик ответов во время прохождения
- система автоматически проверяет результат и сохраняет его
- система сравнивает результат с предыдущей попыткой и формирует уведомления
- студент может подать апелляцию по завершенной попытке, а преподаватель — вернуть решение
- преподаватель анализирует попытки студентов и оставляет комментарий

## 2. Обоснование выбора технологий

### Python

Python выбран как язык разработки, потому что он хорошо подходит для учебных веб-систем, имеет простой синтаксис и зрелую экосистему для серверной разработки.

### Django

Django выбран как основной фреймворк по следующим причинам:

- содержит встроенные механизмы маршрутизации, ORM, шаблонизации, аутентификации и админ-панели
- хорошо соответствует паттерну MVT, который требуется по заданию
- позволяет быстро реализовать устойчивую архитектуру для курсового проекта
- поддерживает тестирование серверной логики стандартными средствами

### Django Templates

Шаблоны Django позволяют реализовать полноценный клиентский слой без отдельного SPA-фронтенда. Это соответствует условиям задания и упрощает интеграцию между представлением и серверной логикой.

### SQLite3

SQLite3 выбрана как встроенная реляционная СУБД, которая:

- не требует отдельной установки сервера БД
- хорошо подходит для учебных и демонстрационных проектов
- полностью поддерживается Django ORM

### VS Code

VS Code указан как рекомендованная среда разработки благодаря поддержке Python, Django, отладки, линтеров и удобной работе с шаблонами.

## 3. Архитектура веб-приложения на основе MVT

Архитектура проекта построена по паттерну MVT:

- `Model` — слой предметных сущностей и логики БД
- `View` — слой серверной логики и обработки HTTP-запросов
- `Template` — слой клиентского представления

### Структурная схема

```text
Пользователь
    |
    v
URL dispatcher
    |
    v
Views / Forms / Services
    |
    +--> Models / ORM / SQLite3
    |
    +--> Templates / HTML / CSS
```

### Модули проекта

- `accounts` — регистрация, авторизация, профиль и роли пользователей
- `testing` — доменные сущности тестирования, серверная логика, аналитика и сервис оценивания
- `templates` — HTML-страницы
- `static/css` — стили пользовательского интерфейса

## 4. Реализация слоя серверной логики

Слой серверной логики построен на class-based views Django и включает:

- домашнюю страницу и каталог курсов
- личную панель пользователя
- регистрацию, вход, выход и редактирование профиля
- CRUD-операции преподавателя для курсов, тестов, вопросов и вариантов ответов
- запуск попытки студентом
- завершение теста и отображение результатов
- просмотр преподавателем всех завершённых попыток

Дополнительно выделен сервис `submit_attempt`, который реализует бизнес-логику оценивания отдельно от представлений. Такое разделение улучшает поддерживаемость и тестируемость.

Для новых сценариев также выделены отдельные сервисы:

- сохранение и восстановление черновика попытки;
- сравнение текущей попытки с предыдущей;
- генерация уведомлений о тестах, объявлениях, отправке попытки, апелляциях и ручной проверке;
- расчет персональных лимитов времени и числа попыток для отдельных студентов.

В приложении также реализованы:

- фильтрация каталога курсов
- подключение к курсу по коду доступа
- публикация объявлений преподавателя
- аналитические панели для преподавателя и студента
- хранение длительности попытки и агрегированных метрик по тестам
- центр уведомлений для обеих ролей

## 5. Реализация слоя логики базы данных

В проекте реализована реляционная модель данных на SQLite3 с помощью Django ORM.

### Основные модели

- `User` — пользователь с ролью преподавателя или студента
- `Course` — учебный курс
- `Announcement` — объявление преподавателя по курсу
- `Enrollment` — запись студента на курс
- `Quiz` — тест внутри курса
- `Question` — вопрос теста
- `Choice` — вариант ответа
- `Attempt` — попытка прохождения теста
- `AttemptDraft` — черновик незавершенной попытки
- `Answer` — ответ студента на конкретный вопрос
- `AttemptReview` — комментарий преподавателя к попытке
- `UserNotification` — уведомление пользователя о событии
- `QuizAccessOverride` — индивидуальные условия прохождения теста для студента
- `AttemptAppeal` — апелляция по завершенной попытке

### Особенности проектирования БД

- ограничение уникальности записи студента на курс
- ограничение уникальности активной попытки по тесту
- ограничение уникальности позиции вопроса внутри теста
- ограничение уникальности позиции варианта ответа внутри вопроса
- хранение итоговых метрик попытки: баллы, процент, число верных ответов
- хранение JSON-черновика ответов для незавершенной попытки
- индексация уведомлений по получателю и статусу прочтения

## 6. Реализация слоя клиентского представления

Клиентский слой реализован через Django Templates и CSS.

Реализованы страницы:

- главная страница
- каталог курсов
- страница курса
- страница теста
- форма прохождения теста
- страница результата
- страница уведомлений
- страница индивидуальных условий по тесту
- страницы подачи и рассмотрения апелляции
- панель пользователя
- страницы авторизации и профиля
- преподавательские формы управления сущностями

Особенности интерфейса:

- простая адаптивная вёрстка без SPA и без перегруженных декоративных элементов
- строгий академический визуальный стиль
- единая навигация между страницами
- панели, таблицы, карточки и аналитические блоки
- визуальное разделение ролей и сценариев
- акцент на читабельность и быструю ориентацию пользователя
- минимальный JavaScript только для автосохранения ответов

- тест регистрации пользователя

- тест автосохранения черновика попытки
- тест восстановления черновика в форме прохождения
- тест сравнения с предыдущей попыткой

- тест выдачи индивидуальных условий по тесту
- тест подачи и рассмотрения апелляции
- тест валидации вопроса с одним правильным ответом

```bash

```
