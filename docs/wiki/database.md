# Слой базы данных

## Выбор базы данных

Для проекта используется `SQLite3`.

Это решение подходит для локальной разработки, демонстрации и тестирования:

- не требует отдельного сервера базы данных;
- хорошо интегрируется с Django;
- упрощает запуск проекта на новой машине;
- делает воспроизведение демо-сценариев быстрым и предсказуемым.

## Основные сущности

### Пользователи и роли

- пользователь;
- роль пользователя: студент или преподаватель;
- профильные поля вроде почты и учебной группы.

### Учебные объекты

- курс;
- запись на курс;
- объявление;
- тест;
- вопрос;
- вариант ответа.

### Попытки и обратная связь

- попытка;
- ответ в попытке;
- отзыв преподавателя;
- черновик попытки;
- апелляция;
- уведомление.

### Дополнительные сущности

- индивидуальные условия по тесту для конкретного студента.

## Связи

- один преподаватель владеет несколькими курсами;
- один курс содержит несколько тестов и объявлений;
- студенты связываются с курсом через запись;
- тест состоит из вопросов;
- попытка принадлежит студенту и тесту;
- ответы попытки связаны с вопросами и выбранными вариантами;
- апелляция относится к конкретной попытке;
- индивидуальные условия привязаны к тесту и студенту.

## Реализация

Основная схема хранится в:

- [accounts/models.py](../../accounts/models.py)
- [testing/models.py](../../testing/models.py)

Для работы с БД используется `Django ORM`, что даёт:

- миграции;
- декларативное описание моделей;
- безопасную работу со связями;
- удобную выборку данных для web и API.

## Почему слой БД соответствует задаче

- структура покрывает весь жизненный цикл тестирования;
- связи отражают реальные отношения между курсами, тестами, студентами и преподавателем;
- дополнительные сущности поддерживают расширенные сценарии, а не только базовый минимум.
